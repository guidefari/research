# 09 — Execution Plans

> **Exam objective:** Understand what an execution plan is, how it is generated, and how to interpret it.

---

## What Is an Execution Plan?

An **execution plan** is Terraform's answer to: *"What will happen if I apply right now?"*

It is generated by `terraform plan` and captures:
- Which resources will be **created**, **updated**, **replaced**, or **destroyed**
- The **reason** for each action
- Attribute **values** (current → new)
- Any values that are `(known after apply)` — computed by the provider after creation

The plan is a **deterministic, repeatable snapshot** of the intended changes.

---

## How Terraform Generates a Plan

```
┌─────────────────────────────────────────────────────┐
│  1. Parse .tf files → desired state                  │
├─────────────────────────────────────────────────────┤
│  2. Read state file → last-known state               │
├─────────────────────────────────────────────────────┤
│  3. Query providers (refresh) → current real state   │
├─────────────────────────────────────────────────────┤
│  4. Build dependency graph                           │
├─────────────────────────────────────────────────────┤
│  5. Walk graph: for each resource:                   │
│     - desired state == current state? → no-op        │
│     - attribute changed, in-place possible? → update │
│     - attribute forces new resource? → replace       │
│     - resource not in config? → destroy              │
│     - resource not in state? → create                │
├─────────────────────────────────────────────────────┤
│  6. Output execution plan                            │
└─────────────────────────────────────────────────────┘
```

## Anatomy of a Plan

### Full Example

```bash
terraform plan
```

```
Terraform used the selected providers to generate the following execution plan.
Resource actions are indicated with the following symbols:
  + create
  ~ update in-place
-/+ destroy and then create replacement

Terraform will perform the following actions:

  # aws_instance.web will be created
  + resource "aws_instance" "web" {
      + ami                          = "ami-0c55b159cbfafe1f0"
      + arn                          = (known after apply)
      + id                           = (known after apply)
      + instance_type                = "t3.micro"
      + private_ip                   = (known after apply)
      + public_ip                    = (known after apply)
      + security_groups              = (known after apply)
      + subnet_id                    = (known after apply)
      + tags                         = {
          + "Environment" = "prod"
          + "Name"        = "web-server"
        }
    }

  # aws_security_group.ssh will be updated in-place
  ~ resource "aws_security_group" "ssh" {
        id          = "sg-0abc123"
        name        = "allow-ssh"
      ~ description = "Old description" -> "New description"
    }

  # aws_s3_bucket.logs must be replaced
  # (forces replacement: bucket name changed)
-/+ resource "aws_s3_bucket" "logs" {
      ~ bucket = "my-old-bucket" -> "my-new-bucket" # forces replacement
        id     = "my-old-bucket"
    }

Plan: 1 to add, 1 to change, 1 to destroy.

──────────────────────────────────────────────────────────────────────────────

Note: You didn't use the -out option to save this plan, so Terraform can't
guarantee that exactly these actions will be performed if you run
"terraform apply" now.
```

---

## Reading "Forces Replacement"

Some attribute changes **cannot** be done in-place — the cloud resource must be destroyed and a new one created. These are annotated with `# forces replacement`:

```
~ bucket = "old-name" -> "new-name"  # forces replacement
```

Common attributes that force replacement:
- EC2: `ami`, `subnet_id`, `availability_zone`
- S3: `bucket` (name)
- RDS: `engine`, `engine_version` (major), `db_name`

> **Exam tip:** When an update forces replacement, the plan shows `-/+` and the summary counts +1 to add AND +1 to destroy.

---

## The Plan Summary Line

```
Plan: 2 to add, 1 to change, 1 to destroy.
```

| Counter | Meaning |
|---|---|
| `to add` | Resources to create |
| `to change` | Resources to update in-place |
| `to destroy` | Resources to delete |

A replacement counts as **both** +1 to add AND +1 to destroy.

## Saving and Using Plan Files

```bash
# Save the plan
terraform plan -out=plan.tfplan

# Inspect the plan (binary → human readable)
terraform show plan.tfplan

# Apply the exact saved plan (no prompt, no re-plan)
terraform apply plan.tfplan
```

Plan files are **binary** — don't try to read them directly. Use `terraform show` to inspect them.

```bash
# Get JSON output of a plan (for automation, parsing)
terraform show -json plan.tfplan | jq '.resource_changes[] | {address: .address, action: .change.actions}'
```

### Why Saved Plans Matter

Without a saved plan:
- `terraform apply` re-runs the plan at apply time
- Infra might change between your `plan` and `apply` — you could apply something different from what you reviewed

With a saved plan:
- `terraform apply plan.tfplan` applies **exactly** what was planned — no drift

> **Best Practice:** Always use `-out` in CI/CD pipelines. The plan stage saves the plan file; the apply stage uses it. This is the gold standard for safe, auditable deployments.

---

## Destroy Plans

```bash
# See what would be destroyed
terraform plan -destroy

# Save a destroy plan
terraform plan -destroy -out=destroy.tfplan

# Apply the destroy plan
terraform apply destroy.tfplan
```

---

## Refresh-Only Plans (Drift Detection)

```bash
# See what drifted without making changes
terraform plan -refresh-only
```

This compares real-world state to your state file and **only** proposes updating the state — it won't modify infrastructure. Use it to detect and review drift.

## Hands-On

```bash
# 1. Plan and read the output symbol by symbol
terraform plan

# 2. Save the plan
terraform plan -out=myplan.tfplan

# 3. Inspect the saved plan
terraform show myplan.tfplan

# 4. Get JSON plan output
terraform show -json myplan.tfplan

# 5. Count actions in a plan
terraform show -json myplan.tfplan | \
  jq '[.resource_changes[].change.actions[]] | group_by(.) | map({(.[0]): length}) | add'
```

---

## Exam Tips

- A plan is read-only — it changes nothing in the real world.
- `terraform apply` always produces a plan before applying (you can see it with `-out` or just before the prompt).
- Plan files are binary — use `terraform show <planfile>` to read them.
- Use `-json` flag with `terraform show` for machine-readable output.
- `-refresh-only` plans are for **drift detection** only — they propose state updates, not infra changes.
- The **plan is not guaranteed** to be applied unchanged unless you use a saved plan file.
